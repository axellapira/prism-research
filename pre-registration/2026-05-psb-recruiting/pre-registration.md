## 1. Summary

This pre-registration locks in the methodology for evaluating
[Prism](https://tryprism.com), an AI-native recruiting agent built by Scrambld,
Inc., on the **Recruiting subset of PeopleSearchBench (PSB)** —
[arXiv:2603.27476](https://arxiv.org/abs/2603.27476) (LessieAI, 2026).

The output of this work will be a technical report (arXiv preprint, cs.IR
category) reporting Prism's scores on the 30 PSB recruiting queries, alongside
the four already-published baselines (Lessie, Juicebox, Exa, Claude Code).

The report will only be published if Prism meets the stopping rules defined in
section 8 of this document.

## 2. Hypothesis

**H1 (primary):** Prism, evaluated on the 30 PeopleSearchBench recruiting
queries using PSB's official Criteria-Grounded Verification harness, achieves a
higher PSB Overall score than **Lessie**, the highest-scoring published
baseline on the recruiting subset and the platform whose authors built the
benchmark.

Lessie's published Overall on the recruiting subset is **68.2 ± 2.8**
(95% CI ≈ [65.4, 71.0]).

**H1 holds if:**
- Prism's Overall score on the recruiting subset exceeds 68.2, AND
- The 95% confidence interval lower bound on Prism's score exceeds 65.4 (the
  lower bound of Lessie's reported CI), AND
- A paired bootstrap test on per-query scores yields p < 0.05 against
  Lessie's per-query scores (per PSB's released results data).

**H2 (secondary, reported regardless):** Prism's Overall score on the
recruiting subset compared to Juicebox (65.7 ± 2.9) and Exa (64.7 ± 3.1).
These comparisons are reported in the paper but are not publication-gating.

Choosing Lessie rather than Juicebox as the primary target is deliberate:
the strongest possible framing of the result is "Prism beats the platform
whose authors built the benchmark." If we anchored on Juicebox, a marginal
result would still leave Lessie at the top of the leaderboard, weakening the
paper's headline. Anchoring on the leader forces a real result.

## 3. Scope

We evaluate Prism on the **30 recruiting queries** (`data/queries/recruiting.jsonl`
in the PSB repository).

We will run Prism on **all 119 PSB queries** internally (Recruiting, B2B
Prospecting, Expert / Deterministic, Influencer / KOL) but only publish the
recruiting subset. Reasoning: Prism is a recruiting product; evaluating it on
the recruiting scenario is the appropriate scope. Internal results on the other
89 queries will be available on request to reviewers and may be released as a
supplementary artifact at our discretion after the recruiting paper is on arXiv.

The full 119-query results are also a hedge: if the recruiting result is
ambiguous, we may expand the scope of the paper rather than augment the query
set with our own queries (which would weaken the third-party-credibility frame).

## 4. System under test

**Prism** is an AI recruiting agent. Architecture relevant to this benchmark:

- A natural-language job description is processed by Prism's matching-profile
  generator to produce role embeddings + structured requirements.
- Prism's sourcing pipeline issues queries against a meta-search across multiple
  external providers, including [Pearch.ai](https://pearch.ai), CoreSignal,
  SerpAPI, and Adzuna. Pearch is one infrastructure component among several;
  this evaluation is not a head-to-head against any single provider but against
  recruiting agents that consume similar infrastructure.
- Returned candidates pass through Prism's three-pass ranker: (1) extraction,
  (2) decomposed sub-scoring with evidence, (3) review-agent reranking on the
  top 30%.
- A wildcard pass surfaces exceptional candidates flagged at weak-match rank.

Prism is invoked exclusively via its public **Model Context Protocol (MCP)**
endpoint at `https://tryprism.com/api/mcp`. This is the same interface a real
recruiter agent uses against Prism in production. We do not invoke any private
or benchmark-specific code path.

## 5. Query adaptation

PSB queries are short natural-language prompts (e.g. "Find backend developers
in London with experience in microservices architecture"). Prism's recruiting
workflow operates on richer job descriptions, as a real recruiter would
provide. To bridge this without disadvantaging Prism by feeding it queries
shaped for a different system:

- For each PSB recruiting query we generate a synthetic 1-paragraph job
  description by passing the query through a single deterministic prompt
  (frozen as `prompts/jd-expansion-v1.md` in this repository).
- Generation parameters: model = `claude-opus-4-7`, temperature = 0,
  no system prompt overrides, no agentic tools.
- The expansion is performed exactly once per query. The 30 expanded JDs are
  saved to `runs/2026-05-XX/expanded_jds.jsonl` and committed to this
  repository before the sourcing run.
- An ablation will be run on a held-out subset of 10 queries (with vs. without
  expansion) and reported as a methodological note.

This adaptation is documented transparently in the paper. Reviewers can re-run
with or without expansion. The expansion step is itself part of how a real
recruiter would interact with Prism — recruiters do not source from one-line
queries.

## 6. Sourcing protocol

For each of the 30 recruiting queries:

1. Generate the synthetic JD per section 5.
2. Via Prism's MCP, create one ephemeral role titled
   `[BENCHMARK] <original PSB query text>`. Wait for matching-profile
   generation to complete.
3. Trigger sourcing via `propose_start_source` → `confirm_start_source` with
   `sourcingProvider = prism` (Prism's full meta-search across providers).
4. Poll `get_source_progress` until the run reaches a terminal state.
5. Pull the top-15 results via `pull_source_results`.
6. Normalize results to PSB's submission CSV schema (section 7).

**Run window.** All 30 sourcing runs will be started within a single
calendar day (UTC). Documentation of exact timestamps for each run is
captured in `runs/2026-05-XX/source_run_logs/`.

**Replication.** Sourcing is run **exactly once per query**. We do not run
multiple sourcing seeds because Prism's blacklist mechanism prevents clean
re-runs against the same role: a second source-run on the same role excludes
already-returned candidates by design, so additional runs would measure
second-tier results rather than ranker variance. This is consistent with the
treatment in Pearch.ai's benchmark (arXiv:2504.02463) and Lessie's
PeopleSearchBench paper, both of which report stochastic uncertainty from
**judge** runs, not sourcing runs.

The sourcing variance question is documented in the paper's limitations
section.

**Cleanup.** All `[BENCHMARK]` roles and all source runs created during this
evaluation will be deleted from the Prism production database after the
evaluation completes and the analysis CSV has been finalized. A small subset
of exceptional candidates surfaced by the benchmark may be moved to the
recruiter's personal talent pool prior to deletion via the standard Prism MCP
flow. This does not alter the evaluation results, which are computed against
the candidate set as-returned by Prism at sourcing time.

## 7. Result normalization

Results are output as a single CSV at
`runs/2026-05-XX/raw_results/prism/recruiting_prism.csv`, conforming to PSB's
submission schema (`docs/submission_guide.md` in the PSB repository):

| Column | Source from Prism |
|---|---|
| `query_id` | PSB query id (e.g. `rec_0001`) |
| `agent_name` | `prism` |
| `name` | Prism candidate name |
| `title` | Current title from candidate experience |
| `company` | Current company from candidate experience |
| `location` | Candidate location |
| `linkedin_url` | LinkedIn URL if present in candidate social_links |
| `email` | Email if present (typically not at sourcing time) |
| `bio` | Prism's match summary / candidate summary |
| `person_data` | The full Prism source-result JSON (raw) |

All normalization logic is committed to this repository as a single Python
script. The script is run exactly once per query against the result of step 5
above and never edited post-hoc.

## 8. Scoring protocol

The PSB scoring harness (`main.py` in the LessieAI/people-search-bench repo)
is run on the normalized CSV.

- **Harness commit hash:** *to be filled in immediately before the run*.
  Expressed as `<repo>@<sha>` in `runs/2026-05-XX/harness_version.txt`.
- **LLM judge model:** `google/gemini-3-flash-preview` (PSB default).
- **Web verification:** Tavily API (PSB default).
- **Concurrency:** 8 (PSB recommendation).

The harness is run **three (3) times** on the same canonical normalized CSV,
each run using a different LLM judge seed via `temperature=0` and an explicit
seed parameter where supported, OR (if seed control is not exposed) by relying
on the LLM provider's run-to-run variance. Per-query scores from each run are
saved separately.

**Statistical reporting:**

- Each run produces per-query Relevance Precision, Effective Coverage, and
  Information Utility scores per PSB's published methodology.
- We report the **mean across the 3 runs** as the primary score for each
  metric, with **95% confidence intervals from a 1000-iteration paired
  bootstrap** over the 30 queries × 3 runs (90 query-level scores per metric).
- The **paired bootstrap test** for H1 compares Prism's per-query scores to
  Juicebox's per-query scores from PSB's released results, using the same
  bootstrap procedure described in PSB's paper (Smucker, Allan, Carterette,
  2007 sampling). Significance threshold: **p < 0.05 (two-sided)**.
- Multiple-comparison correction: we are testing one primary hypothesis (H1
  vs. Juicebox). H2 (vs. Lessie) is exploratory and reported without
  Bonferroni adjustment, with a clear note that it is exploratory.

## 9. Stopping rules and publication criteria

Locked in advance. We will not change these after seeing results.

**G1 — smoke test (post 5-query subset):** if the adapter pipeline produces
output that PSB's verifier rejects (e.g. malformed CSV, schema mismatch), we
fix and rerun the smoke test. No statistics are computed yet.

**G2 — post-sourcing-run, before scoring:** the canonical CSV is committed
to `runs/2026-05-XX/raw_results/prism/` before we run any scoring. Once
scoring begins, the CSV is frozen.

**G3 — post-scoring (the publication gate):**

The threshold is **Lessie's published Overall of 68.2** on the recruiting
subset. Three pre-committed outcomes:

- **Publish as "leads PSB recruiting"** if and only if Prism's mean Overall
  exceeds 68.2 AND the paired bootstrap test against Lessie's per-query
  scores gives p < 0.05.
- **Publish as "comparable to top of PSB recruiting leaderboard"** if Prism's
  mean Overall is within ±2 points of 68.2 (i.e. between 66.2 and 70.2) AND
  the bootstrap test against Lessie gives p ≥ 0.05. The paper will report
  the result honestly — Prism is statistically tied with Lessie — and frame
  Prism's standing relative to all four baselines.
- **Do not publish** if Prism's mean Overall is more than 2 points below
  Lessie's 68.2 (i.e. < 66.2).

The choice of Lessie as the threshold is locked here regardless of the H2
results — even if Prism beats Juicebox by a wide margin, that alone is not
sufficient to publish under this pre-registration. The paper either claims a
top-leaderboard result (H1 met) or makes no comparable-to-leader claim at
all. Anchoring high removes the temptation to retreat to a weaker target if
the data is borderline.

## 10. Reporting

Regardless of outcome, the following will be made public via this repository:

- The 30 expanded JDs.
- The canonical CSV of Prism's results.
- The 3 scoring-run JSON outputs from the PSB harness.
- The bootstrap analysis script and its output tables.
- A per-query error analysis: for every query where Prism scored below the
  median of the four PSB baselines, a short writeup of the failure mode.

Privacy: per the PSB convention, **raw per-person results in the CSV will be
anonymized** before public release (LinkedIn URLs preserved; names redacted
to first-name + last-initial; emails removed). The non-anonymized CSV will be
preserved internally for audit but will not be public.

## 11. Limitations declared in advance

These will be in the paper's limitations section regardless of outcome:

- **Small query set.** 30 queries is a thin sample. We cite PSB's own
  acknowledgement of this, defer to their query-design rationale, and do not
  augment.
- **Web-verification coverage bias.** PSB's Tavily-based verification is
  biased toward already-indexed candidates. A genuinely fitting candidate
  with minimal web presence may be marked unverified. This affects all
  platforms equally but caps the achievable ceiling for any system that
  surfaces hard-to-find people.
- **Pearch as infrastructure.** Prism uses Pearch.ai as one component of its
  sourcing pipeline. We are not benchmarking against Pearch directly. The
  evaluation measures Prism's full system (meta-search + ranker + reranker
  + wildcard pass) against AI recruiting agents that consume similar
  infrastructure. The paper will be explicit on this point.
- **Sourcing-run variance not captured.** As discussed in section 6,
  Prism's blacklist mechanism prevents clean re-runs. The 3-seed scoring is
  judge-variance only.
- **Recruiting-only scope.** Generalization to B2B / Expert / Influencer is
  not claimed; internal results available on request.
- **Self-evaluation risk.** This benchmark is run by the system's authors.
  Mitigations: (a) pre-registration timestamped via this committed
  document, (b) PSB's verification pipeline is third-party (we do not modify
  the harness), (c) all artifacts including failures are public.
